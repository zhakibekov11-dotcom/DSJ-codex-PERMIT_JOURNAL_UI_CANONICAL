# PERMIT_JOURNAL_UI_CANONICAL — Brief

## Purpose

Эта ветка инициализирует контекст для реализации модуля "Журнал допусков / Permit Journal" в большом контуре DSJ.

Модуль нужен для цифрового учёта и управления допусками, нарядами-допусками, подрядными актами-допусками и блокирующими проверками перед работами.

## Product position

- Универсальная государственная форма "Журнал допусков" для всех случаев не выявлена.
- Permit Journal является цифровой надстройкой над несколькими нормативными процессами.
- Для UI используется "Журнал допусков".
- Для архитектуры используется PermitJournal / PermitEntry.
- Главный MVP-сценарий - наряд-допуск на работы повышенной опасности.
- Второй важный сценарий - contractor access / акт-допуск подрядчика.
- Проверки обучения, инструктажа, удостоверений, медосмотра и СИЗ являются блокирующими перед активным допуском.

## What future Codex execution will implement

Будущая реализация должна создать интерфейс и системную связку для:

- списка допусков;
- создания PermitEntry;
- редактирования до подписания;
- precheck;
- согласований;
- подписей / ЭЦП evidence;
- активации допуска;
- приостановки;
- закрытия;
- архива;
- audit trail;
- PDF/export;
- связей с сотрудниками, подрядчиками, удостоверениями, обучением, инструктажами, документами и аудитом.

## Hard boundaries

- Не реализовывать отраслевые P1/P2 формы в MVP.
- Не выдумывать новые типы допусков.
- Не выдумывать новые Adilet-акты.
- Не менять legal meaning терминов.
- Не хранить диагнозы в PermitEntry.
- Не делать signed fields редактируемыми после approved.
- Не считать PDF источником подписанной истины.
- Не хардкодить один срок хранения для всех документов.
